# ADR-002: Pure ESM — ES Modules Throughout

**Status:** Accepted  
**Date:** 2026-03-26

## Context

Node.js ESM support is stable in Node 18+. CJS interop is possible via `createRequire` but adds friction — dual-module hazard, `__filename`/`__dirname` workarounds, and subtle differences in singleton semantics. CJS is the legacy format; ESM is the direction the ecosystem is moving.

## Decision

All packages use `"type": "module"` in `package.json`, `import`/`export` syntax throughout, and explicit `.js` extensions on all local imports (required by the ESM spec). `createRequire` shims are used only for CJS-only third-party packages that have not yet published an ESM build (currently: `rhea`, `amqplib`, `node-zookeeper-client`).

## Consequences

**Positive:**
- Future-proof — no dual-module hazard, no CJS-to-ESM upgrade debt.
- Works natively in Node 22+ without flags.
- Top-level `await` available in all modules.

**Negative:**
- Some third-party CJS-only packages require a `createRequire` shim, adding a small amount of boilerplate per affected component.
- Packages cannot be imported with `require()` by callers still on CJS. Callers must use ESM or dynamic `import()`.
