# ADR-003: npm Workspaces Monorepo — Per-Component Packages

**Status:** Accepted  
**Date:** 2026-03-26

## Context

Apache Camel publishes each component as an independent Maven artifact (`camel-direct`, `camel-seda`, `camel-http`, etc.). This allows users to install only the components they need and keeps the dependency graph minimal. We want the same model in npm. A monorepo is necessary to develop all components together while keeping them independently publishable.

## Decision

The repository is an npm workspaces monorepo. Each component is an independent package under `packages/` with its own `package.json`, `exports` map, `README.md`, and test suite. `camel-lite-core` is independently installable and carries no component dependencies. Users compose their own runtime by installing the components they need.

## Consequences

**Positive:**
- Mirrors Camel's modular artifact model — the conceptual mapping to Camel is direct.
- Users install only the components they need, keeping `node_modules` lean.
- Component boundaries are enforced by `package.json` — cross-component imports that bypass the exports map fail with `ERR_PACKAGE_PATH_NOT_EXPORTED`.

**Negative:**
- More `package.json` files to maintain (one per component).
- Workspace symlinks can cause confusion if a package is missing an `exports` field — the import resolves via symlink but Node's exports map enforcement still applies.
