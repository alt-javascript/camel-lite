# ADR-014: Leader Election — Pluggable LockStrategy with Three Backends

**Status:** Accepted  
**Date:** 2026-03-27

## Context

Apache Camel's `master:` component uses ZooKeeper or Kubernetes leases for leader election. In a JavaScript context, ZooKeeper requires a native client (`node-zookeeper-client`, CJS-only) which works but adds a dependency. A file-lock mechanism using `O_EXCL | O_CREAT` exclusive file creation has zero external dependencies and covers single-host deployments. Consul provides distributed coordination via HTTP (native `fetch`) for multi-host deployments without ZooKeeper infrastructure.

## Decision

The `master:` component uses a pluggable `LockStrategy` interface. Three concrete implementations are provided:

| Backend | Class | Dependency | Use case |
|---------|-------|------------|----------|
| `file` (default) | `FileLockStrategy` | none | Single-host deployments |
| `zookeeper` | `ZooKeeperStrategy` | `node-zookeeper-client` (CJS shim) | Multi-host with existing ZooKeeper |
| `consul` | `ConsulStrategy` | none (native `fetch`) | Multi-host with Consul |

The backend is selected via URI parameter: `master:serviceName?backend=file|zookeeper|consul`.

**Important limitation:** `FileLockStrategy` is **not safe on NFS shares**. `O_EXCL | O_CREAT` file creation is advisory on many NFS implementations — two nodes can simultaneously succeed in creating the lock file. For distributed deployments (multiple hosts sharing a network filesystem), use the `zookeeper` or `consul` backend.

## Consequences

**Positive:**
- Works out of the box with no external dependencies (file backend).
- Extensible: additional backends can be added by implementing the `LockStrategy` interface.
- Consul backend requires no npm dependency beyond Node's built-in `fetch`.

**Negative:**
- File backend is single-host only and carries an NFS caveat that users must be aware of.
- Three backends to maintain and test.
