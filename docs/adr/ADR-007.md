# ADR-007: Component Factory Chain — Component → Endpoint → Producer/Consumer

**Status:** Accepted *(not revisable)*  
**Date:** 2026-03-26

## Context

Apache Camel uses a three-tier factory chain as its core extension model: a `Component` creates `Endpoint` instances from URIs; an `Endpoint` creates `Producer` instances (for `to:` steps) and `Consumer` instances (for `from:` steps). This separation puts URI configuration parsing in the `Endpoint` and message dispatch logic in the `Producer`/`Consumer`, and allows the same `Endpoint` configuration to serve both directions.

## Decision

camel-lite follows the same factory chain exactly:

```
Component.createEndpoint(uri, remaining, params, context)
  → Endpoint.createProducer()
  → Endpoint.createConsumer(pipeline)
```

URI parsing happens in the `Endpoint` constructor. The scheme is registered on `CamelContext` via `addComponent(scheme, component)`. Route compilation calls `component.createEndpoint()` for each URI encountered, and the resulting endpoint creates the appropriate `Producer` or `Consumer`.

## Consequences

**Positive:**
- Direct conceptual port of Camel's proven architecture — the mental model transfers from Camel documentation.
- Component authors have a clear, stable contract: implement three classes with well-defined responsibilities.
- `CamelContext` is component-agnostic — it routes scheme lookups through the registry without knowing anything about individual components.

**Negative:**
- Three classes per component adds boilerplate, even for simple components.
- Some components (e.g. `direct:`) would be implementable with fewer layers, but consistency with the pattern is preferred.
