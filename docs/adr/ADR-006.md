# ADR-006: Expression Language — Native Functions + simple()/js() Wrappers

**Status:** Accepted  
**Date:** 2026-03-26

## Context

Apache Camel uses a string-based Simple expression language for predicates, filters, and log messages. In JavaScript, native functions are more natural, more powerful, and debuggable with standard tooling. However, YAML-based route definitions cannot embed native functions — they need a string expression form.

## Decision

Three expression forms are supported. All normalise internally to `(exchange) => value`:

1. **Native JS function** — passed directly:
   ```js
   .filter(ex => ex.in.body.length > 0)
   ```

2. **`simple(expr)`** — compiled subset of Camel Simple language:
   ```js
   .filter(simple('${body.length} > 0'))
   ```

3. **`js(code)`** — raw JS string evaluated with `new Function('exchange', code)`:
   ```js
   .filter(js('return exchange.in.body.length > 0'))
   ```

The `simple()` compiler supports: `${body}`, `${header.X}`, `${exchangeProperty.X}`, `${out.body}`, logical operators (`&&`, `||`), `contains`, `regex`, and `==` mapped to `===`.

**Known limitation:** `simple()` does not support string interpolation with surrounding literal text (e.g. `'Hello ${body}'`). The compiler replaces `${body}` inline and the resulting expression is not valid JavaScript when literal text is present. Use `js()` or a `process()` step with a template literal for that case.

## Consequences

**Positive:**
- Idiomatic JavaScript — native functions are the primary form with no overhead.
- String expressions allow YAML route definitions to carry logic without embedded JS.
- All three forms share the same normalised function interface — processors and predicates don't distinguish between them.

**Negative:**
- `simple()` is a subset of Camel Simple — not all Camel Simple expressions are supported.
- The string interpolation limitation means some Camel YAML patterns need a `process()` workaround.
