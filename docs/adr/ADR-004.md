# ADR-004: No DI Container in Core Components

**Status:** Accepted *(not revisable)*  
**Date:** 2026-03-26

## Context

Several JavaScript DI frameworks exist: InversifyJS, tsyringe, awilix, and others. Using one would tightly couple `camel-lite-core` to a specific container and its lifecycle model, complicate unit testing (mock injection patterns differ per container), and pull in transitive dependencies for every user — including those who have no interest in DI.

## Decision

`camel-lite-core` and all component packages contain no DI container. `CamelContext` is a plain JavaScript object with an explicit `addComponent(scheme, component)` / `addRoutes(builder)` API. Components and routes are wired manually or via test helpers. Container integration is provided separately through the boot starters (`boot-camel-lite-starter`, `boot-camel-lite-extras-starter`), which are optional packages.

## Consequences

**Positive:**
- `camel-lite-core` is usable without any DI framework — install and go.
- Unit testing is straightforward: construct objects, call methods, assert outcomes.
- No container compatibility matrix to maintain as container versions evolve.

**Negative:**
- Users who want CDI-managed lifecycle (auto-start, config binding, conditional beans) must install and configure the boot starters.
- No auto-wiring in the core — every component must be explicitly registered.
