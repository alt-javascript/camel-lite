# ADR-012: Boot Integration — CDI Conditional Beans and dependsOn Ordering

**Status:** Accepted  
**Date:** 2026-03-27

## Context

`boot-camel-lite-starter` registers component beans conditionally based on `boot.camel-lite.<scheme>.enabled`. CDI's `dependsOn` mechanism resolves dependencies at wiring time and throws if a named bean does not exist (because its condition evaluated to false). If `camelLiteContext.dependsOn` listed all component bean names, disabling any one component would break context initialisation.

## Decision

`camelLiteContext.dependsOn` is set to `['routeRegistry']` only — it does not list individual component beans. Instead, `RouteRegistry.applyComponents()` scans `applicationContext` at runtime for all beans whose name matches the `camelComponent.*` prefix, and registers the ones it finds. This scan is inherently tolerant of missing beans: if a component bean was not registered (condition false), it simply is not present in the scan result.

Component beans are declared before `routeRegistry`, and `routeRegistry` before `camelLiteContext`, in the CDI config array. CDI's topological sort handles beans with explicit `dependsOn`; for beans without it, declaration order is used as the sort key.

## Consequences

**Positive:**
- Any subset of components can be disabled via config without causing CDI initialisation errors.
- Adding a new component to the extras starter does not require updating `dependsOn` lists anywhere.

**Negative:**
- Beans without explicit `dependsOn` rely on declaration order in the config array for initialisation ordering. Reordering the array without understanding this dependency could cause subtle failures.
