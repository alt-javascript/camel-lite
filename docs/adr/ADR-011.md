# ADR-011: Route Loading — Four Entry Points for Different Input Sources

**Status:** Accepted  
**Date:** 2026-03-27

## Context

Route definitions can arrive from multiple sources with different parsing requirements: files on disk (need I/O + extension detection), stdin streams (need draining + format detection), config systems that pre-parse YAML (already a JS object — double-parsing would fail), and in-memory strings (need format detection). A single `load(x)` function with overloaded semantics would produce confusing behaviour; explicit entry points make the contract clear.

## Decision

`RouteLoader` provides four static entry points:

| Method | Input | Behaviour |
|--------|-------|-----------|
| `loadFile(path)` | File path string | Reads file from disk, sniffs format from extension (`.yaml`/`.yml`/`.json`); unknown extensions fall back to content-sniff. Delegates to `loadString`. |
| `loadString(text, format?)` | String | Parses YAML or JSON. Auto-detects format if `format` is omitted by checking for `{` or `[` as the first non-whitespace character. |
| `loadStream(stream)` | Readable stream | Drains the stream to a string, then delegates to `loadString` with content-sniff. Used for stdin (`-r -`). |
| `loadObject(obj)` | JS object | Accepts an already-parsed object and skips parsing entirely. Used by boot starters where `@alt-javascript/config` deserialises YAML at load time. |

All four methods return a normalised route definition array.

## Consequences

**Positive:**
- Covers all practical input sources — file, stdin, config, and in-memory.
- `loadObject` prevents the double-parse bug that would occur if a boot starter passed a pre-parsed object to `loadString`.
- Each entry point has a single clear responsibility.

**Negative:**
- Four entry points to document, test, and maintain rather than one polymorphic loader.
