# ADR-008: URI Parsing — Manual Split for Path, URLSearchParams for Query

**Status:** Accepted  
**Date:** 2026-03-26

## Context

The obvious approach to parsing a URI like `log:MyLogger?level=INFO` is `new URL('log://MyLogger?level=INFO')`. However, the URL spec normalises the hostname portion to lowercase. For components where the URI path segment is case-sensitive — `log:MyLogger` (logger category name) and `master:MyService` (service name used in lock file paths) — `new URL(...)` silently corrupts the data without throwing an error. This is a silent correctness bug.

## Decision

URI path/name is extracted by splitting on `?` and taking the left portion, then stripping the scheme prefix manually. Only the query portion (everything after the first `?`) is passed to `URLSearchParams` for parameter extraction. The `URL` constructor is not used for camel-lite URI parsing anywhere in the codebase.

```js
const [schemeAndPath, query = ''] = uri.split('?');
const remaining = schemeAndPath.slice(schemeAndPath.indexOf(':') + 1);
const params = Object.fromEntries(new URLSearchParams(query));
```

## Consequences

**Positive:**
- Case-preserving — `log:MyLogger` round-trips correctly.
- No URL spec ambiguity or hostname-normalisation surprises.

**Negative:**
- Manual split is less robust than a full URL parser for edge cases (URIs with multiple `?`, encoded characters in the path, etc.). These cases do not arise in practice for camel-lite URI syntax.
