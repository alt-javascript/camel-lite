# ADR-001: Plain JavaScript — No TypeScript

**Status:** Accepted  
**Date:** 2026-03-26

## Context

The project is a JavaScript port of Apache Camel patterns. TypeScript would add a type-system translation layer that could distort idiomatic Camel designs — many Camel patterns rely on duck typing and runtime polymorphism that TypeScript's structural type system would fight. TypeScript also adds a mandatory build step (`tsc`) and, for DI-heavy Camel-style patterns, introduces decorator dependencies.

## Decision

All source is plain JavaScript (`.js`). JSDoc annotations are acceptable for editor hints and API documentation. No TypeScript source files, no `tsc` compilation step, no `tsconfig.json` in component packages.

## Consequences

**Positive:**
- No build step — source files run directly under Node.js.
- Code is readable by anyone familiar with JavaScript without TypeScript knowledge.
- No type-system friction when porting Camel patterns that rely on duck typing and dynamic dispatch.

**Negative:**
- No compile-time type checking. Type errors surface at runtime or in tests.
- Public API contracts must be communicated via JSDoc comments and test coverage rather than type signatures.
